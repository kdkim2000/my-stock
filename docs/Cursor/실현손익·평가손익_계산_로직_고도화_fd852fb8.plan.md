# 실현손익·평가손익 계산 로직 고도화

## 요구사항 정리

- **거래일 기준**: 매매 내역을 **거래일(날짜) 순**으로 처리.
- **수량 0이 되면**: 매수수량과 매도수량의 합(순수량)이 0이 되는 시점에서 해당 구간의 실현손익(차액)을 확정하고, **이후 매수부터 새로 매수단가를 잡음**.
- **매도 시**: 그 시점의 **보유분 매수 평균가**로 매도 원가를 계산.
- **추가 매수 시**: (남은 수량 × 기존 매수 평균가) + (추가 매수 수량 × 매수가)를 합쳐 **가중 평균**으로 새 매수 평균가 계산.

즉, **날짜 순 + 종목별 보유 수량/보유 원가(buyQty, buyValue) 상태기계**로, 매수 시 누적·매도 시 비례 차감·보유 수량 0이면 원가 리셋(다음 매수부터 새 원가)이 되어야 함.

## 현재 구현과의 차이


| 구분                                                                   | 현재                                                                                                                                                          | 변경 후                                                                                                                                          |
| -------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **실현손익** [lib/analysis.ts](lib/analysis.ts)                          | 이미 날짜 순, 매도 시 비례 원가 차감·매수 시 누적. 수량 0이 되면 자연스럽게 buyQty/buyValue=0으로 다음 매수부터 새 원가.                                                                            | 동일 알고리즘 유지. 같은 날 여러 건일 때 정렬 보강(날짜 같으면 행 순서 등), 주석으로 “거래일 기준·수량 0 시 차액 확정·가중평균” 명시.                                                            |
| **평가손익(보유 원가)** [lib/portfolio-summary.ts](lib/portfolio-summary.ts) | **날짜 순이 아님**. 종목별로 “총 매수량/총 매수금액”과 “총 매도량”만 합산해, netQty와 `buyValue * (netQty / buyQty)`로 보유 원가 계산. → 전량 매도 후 재매수한 경우 **잔여 수량의 원가가 잘못됨**(이전 매수까지 포함한 비례 배분). | **날짜 순 처리**로 변경. 분석과 동일하게 buyQty/buyValue를 날짜 순으로 갱신하고, **최종 buyQty > 0인 종목만** position으로 두며, **buyAmount = 최종 buyValue**(현재 보유분의 매수 원가)로 사용. |


## 구현 계획

### 1. [lib/portfolio-summary.ts](lib/portfolio-summary.ts) — 보유 원가를 거래일 순으로 계산

- `computePortfolioSummaryFromTransactions` 내부에서:
  - `transactions`를 **날짜 순 정렬** (예: `sortByDate`와 동일한 규칙, [lib/analysis.ts](lib/analysis.ts)의 `sortByDate` 참고).
  - 종목별 상태: `buyQty`, `buyValue`만 유지 (매도 시 비례 차감, 0 이하로 떨어지면 0으로 클램프).
  - **매수**: `buyQty += q`, `buyValue += price * q`.
  - **매도**: `costOfSold = buyQty > 0 ? (buyValue * q) / buyQty : 0`, `buyQty -= q`, `buyValue -= costOfSold`, 이후 `buyQty`/`buyValue`를 0 미만이면 0으로.
  - 루프 종료 후 `buyQty > 0`인 종목만 수집:
    - `quantity = buyQty`
    - `buyAmount = buyValue` (현재 보유분의 매수 원가 = 가중평균·수량 0 리셋 반영)
  - `totalBuyAmount`는 위 position들의 `buyAmount` 합.
- 정렬은 `portfolio-summary` 내부에서 구현하거나, analysis의 `sortByDate`를 공통 유틸로 빼서 재사용해도 됨.

### 2. [lib/analysis.ts](lib/analysis.ts) — 실현손익 로직 명시 및 같은 날 정렬 보강

- **동작 변경 최소화**: 이미 “거래일 순 + 매도 시 당시 매수 평균가로 원가 계산 + 추가 매수 시 가중평균”이 구현되어 있음.
- **같은 날 여러 건**: 시트에 시간이 없으므로, 같은 날짜 내 순서는 **원본 배열 인덱스**로 유지하면 됨. 정렬 시 `(Date, index)` 2차 키로 정렬해 일관성 유지.
- **주석 보강**: `computeAnalysis` 상단에 “거래일 기준, 수량 0이 되면 차액 확정 후 다음 매수부터 새 매수단가, 매도 시 당시 보유분 평균가, 추가 매수 시 가중평균”을 명시.

### 3. [lib/analysis.ts](lib/analysis.ts) — computeCumulativePnl

- 이미 날짜 순으로 같은 상태기계(buyQty, buyValue)를 사용하므로 **로직 변경 없음**.
- 같은 날 정렬을 `computeAnalysis`와 동일한 규칙(날짜 + 인덱스)으로 맞추면, 일자별 실현손익 합계가 분석과 일치해 일관성 유지.

### 4. 공통 정렬

- **옵션 A**: [lib/analysis.ts](lib/analysis.ts)의 `sortByDate`를 `lib/sort-transactions.ts` 등으로 분리하고, analysis와 portfolio-summary에서 공통 사용. 같은 날은 원본 인덱스로 2차 정렬.
- **옵션 B**: 각 모듈에서 `[...transactions].sort((a,b) => Date(a.Date) - Date(b.Date) || indexOf(a) - indexOf(b))` 형태로 동일 규칙 적용.

한 곳에서만 정렬 규칙을 두는 쪽(옵션 A)이 유지보수에 유리함.

## 결과 정리

- **실현손익**: 거래일(및 동일일 내 순서) 기준, 수량 0이 되면 차액 확정·이후 매수부터 새 단가, 매도 시 당시 평균가·추가 매수 시 가중평균이 코드와 주석으로 명확해짐.
- **평가손익(보유 원가)**: 거래일 순 처리로 “현재 보유 수량에 대한 매수 원가”가 전량 매도 후 재매수한 경우에도 올바른 가중평균(및 리셋)으로 계산됨.
- **누적 실현손익 차트**: 동일 정렬·동일 상태기계로 분석과 일치 유지.
