# 평가손익을 계산하지 못하는 원인 단계별 파악

## 데이터 흐름 개요

```mermaid
flowchart LR
  subgraph sheet [시트]
    Tx[매매내역 시트]
    Master[종목코드 마스터]
    Agg[종목별 집계]
  end
  subgraph api [API]
    GET["GET /api/kis/portfolio-summary"]
    getTx[getTransactions]
    compute[computePortfolioSummaryFromTransactions]
    codeMap[getTickerCodeMap]
    enrich[enrichPortfolioSummaryWithKis]
    getPrice[getCurrentPrice]
  end
  subgraph ui [UI]
    Table[TickerAnalysisTable]
  end
  Tx --> getTx
  getTx --> compute
  Master --> codeMap
  Agg --> codeMap
  compute --> enrich
  codeMap --> enrich
  enrich --> getPrice
  getPrice --> enrich
  enrich --> GET
  GET --> Table
  Table -->|positionsByTicker[r.ticker]| pos[position.profitLoss]
```



평가손익(`profitLoss`)은 **포지션 단위**로 `enrichPortfolioSummaryWithKis`에서 계산되고, UI는 **analysis의 ticker**와 **portfolio의 position**을 `r.ticker === p.ticker`로 매칭해 `pos.profitLoss`를 표시한다.

---

## 단계 1: 포지션 자체가 없는 경우 (positions에 종목이 없음)

**위치**: [lib/portfolio-summary.ts](lib/portfolio-summary.ts) `computePortfolioSummaryFromTransactions`

- **조건**: `netQty = buyQty - sellQty <= 0` 이면 해당 종목은 `positions`에 넣지 않음.
- **결과**: 종목별 분석 테이블에는 해당 ticker가 **analysis**에는 있지만(과거 매매 이력), **portfolio.data.positions**에는 없음 → `positionsByTicker[r.ticker]`가 `undefined` → `quantity`·`profitLoss` 모두 0.
- **원인**: 매수 총량과 매도 총량이 같거나, 매도가 더 많아 **잔고 수량이 0 이하**로 계산된 경우. 이 경우는 “잔고 수량이 있는데”와 모순되므로, 사용자가 말하는 “잔고 수량이 존재”는 **다른 종목**이거나 **다른 데이터 소스(시트/캐시)**와의 불일치 가능성을 의미할 수 있음.

---

## 단계 2: 분석 ticker와 포트폴리오 position의 키 불일치

**위치**: [components/dashboard/TickerAnalysisTable.tsx](components/dashboard/TickerAnalysisTable.tsx) `positionsByTicker[r.ticker]`

- **분석 쪽 키**: [lib/analysis.ts](lib/analysis.ts)에서 `t = row.Ticker?.trim() || ""` 로 **trim된 Ticker**.
- **포트폴리오 쪽 키**: [lib/portfolio-summary.ts](lib/portfolio-summary.ts)에서 `t = (row.Ticker || "").trim()` 로 **동일하게 trim된 Ticker**.
- **불일치 가능성**: 시트에 **같은 종목이 서로 다른 문자열**로 들어간 경우 (예: "삼성전자" vs "삼성전자 ", "005930" vs "005930.KS"). 한쪽은 analysis에만, 다른 쪽은 portfolio에만 들어가면 **보유수량은 다른 행에서 보이고 평가손익은 해당 행에서 0**으로 나올 수 있음. 또는 **전각/반각 공백**, **대소문자** 등으로 키가 달라지면 매칭 실패.

---

## 단계 3: 종목코드 매핑 실패 (code가 undefined)

**위치**: [lib/portfolio-summary.ts](lib/portfolio-summary.ts) `resolveTickerToCodeWithMap(trimmed, codeMap)` → `normalizeTickerCode(code)`

- **resolveTickerToCodeWithMap**  
  - ticker가 **정확히 6자리 숫자**면 그대로 코드로 사용.  
  - **앞 6자리가 숫자**면 그 6자리만 추출 (예: "005930.KS" → "005930").  
  - 그 외에는 `codeMap[t]`만 사용.
- **codeMap**은 [lib/ticker-mapping.ts](lib/ticker-mapping.ts) `getTickerCodeMap()`에서 **(1) 종목코드 마스터 시트 (2) 종목별 집계 시트 Code 컬럼 (3) 하드코딩 TICKER_TO_CODE** 순으로 채워짐.
- **원인**  
  - 한글/영문 ticker(예: "NAVER", "카카오")가 **마스터·집계 시트·하드코딩 어디에도 없음**.  
  - 시트의 Ticker가 **공백/특수문자**로 마스터·집계의 Ticker와 미세하게 달라서 codeMap 키와 불일치.  
  - 마스터/집계 시트 미설정 또는 읽기 실패로 codeMap이 비어 있고, 해당 ticker가 하드코딩에도 없음.
- **결과**: `code`가 undefined → `getCurrentPrice` 호출 안 함 → `currentPrice = null` → `marketValue = p.buyAmount`, `profitLoss = 0`.

---

## 단계 4: 종목코드 정규화 실패 (normalizeTickerCode가 undefined 반환)

**위치**: [lib/portfolio-summary.ts](lib/portfolio-summary.ts) `normalizeTickerCode(code)`

- **동작**: 공백 제거 후, **정확히 6자리 숫자**면 통과, **숫자만 있고 6자리 이하**면 앞 0 채워 6자리로 만듦. 그 외(문자 포함, 7자리 이상 등)는 `undefined`.
- **원인**: 시트/마스터에 **"005930-1"**, **"KR7005930000"** 처럼 문자가 섞였거나 6자리가 아닌 코드가 들어온 경우.
- **결과**: `code`가 undefined로 정리됨 → 현재가 미조회 → `profitLoss = 0`.

---

## 단계 5: KIS 현재가 조회 실패 (getCurrentPrice가 null 반환)

**위치**: [lib/kis-api.ts](lib/kis-api.ts) `getCurrentPrice(tickerCode)`

- **진입 전**: 인자가 6자리 숫자가 아니면 즉시 `null` 반환.
- **토큰**: `getAccessToken()` 실패 (KIS_APP_KEY/SECRET 미설정, 토큰 발급 실패) → `null`.
- **API 호출**: `res.ok`가 false (4xx/5xx, 네트워크 오류) → `null`.
- **응답 파싱**: `rt_cd` 또는 `msg_cd`가 "0"이 아님(API 오류) → `null`. `output.stck_prpr` 없음/빈값 → `null`.
- **원인**: KIS 미설정, 장 마감/권한/종목코드 오타, 일시적 API 오류 등.
- **결과**: `currentPrice = null` → `marketValue = p.buyAmount`, `profitLoss = 0`.

---

## 단계 6: UI에서의 매칭 및 표시

**위치**: [components/dashboard/TickerAnalysisTable.tsx](components/dashboard/TickerAnalysisTable.tsx)

- `positionsByTicker`는 `portfolio.data.positions`를 `p.ticker`를 키로 한 맵으로 만듦.
- 각 행은 `analysis.data.tickers`를 기준으로 하고, `pos = positionsByTicker[r.ticker]`, `profitLoss = pos?.profitLoss ?? 0`.
- **정상**: 포지션이 있고 위 단계 1~5가 모두 성공하면 해당 행에 평가손익이 표시됨.
- **비정상**: 단계 1~5 중 어느 하나라도 실패하면, 그 종목에 대해 **보유수량은 있더라도**(다른 경로로 수량이 보이거나, 사용자 인지와 다른 종목) **평가손익은 0**으로 나올 수 있음.

## 요약: 원인별 체크 포인트


| 단계  | 확인 위치                                            | 확인 내용                                    |
| --- | ------------------------------------------------ | ---------------------------------------- |
| 1   | 시트 매매내역, computePortfolioSummaryFromTransactions | 해당 종목의 netQty(buyQty−sellQty)가 실제로 > 0인지 |
| 2   | 시트 Ticker vs analysis/portfolio 키                | 동일 종목이 동일한 문자열(trim, 공백, 접미사)로 들어가는지     |
| 3   | getTickerCodeMap, 마스터/집계 시트·하드코딩                 | 해당 ticker에 대한 6자리 코드가 codeMap에 들어가는지     |
| 4   | 마스터/집계 시트 Code 컬럼                                | 코드가 숫자 6자리 또는 숫자만 6자리 이하(앞 0 보정 가능)인지    |
| 5   | KIS env, getAccessToken, inquire-price 응답        | 토큰 발급·API 호출·응답 rt_cd·stck_prpr이 정상인지    |


이 순서대로 로그 또는 조건 분기로 어느 단계에서 끊기는지 확인하면, “평가손익을 계산하지 못하는” 구체적인 원인을 좁힐 수 있다.
