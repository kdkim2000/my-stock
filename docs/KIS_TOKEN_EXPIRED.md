# KIS API "기간이 만료된 token" 오류 조치 가이드

## 오류 메시지 예시

```
[KIS] 단계5: inquire-price HTTP 실패 code=001440 status=500 body={"rt_cd":"1","msg1":"기간이 만료된 token 입니다.","msg_cd":"EGW00123"}
```

## 원인 분석

| 항목 | 설명 |
|------|------|
| **의미** | 한국투자증권(KIS) Open API 접근 토큰이 **이미 만료**된 상태에서 현재가(inquire-price) API를 호출한 경우 발생합니다. |
| **msg_cd** | `EGW00123` = 토큰 만료 관련 오류 코드 |
| **발생 상황** | 1) 앱이 캐시한 토큰의 만료 시각을 잘못 계산했을 때(예: KST/UTC 해석 오차) 2) 토큰 유효 시간(24시간)이 지났는데 같은 서버 인스턴스가 캐시를 계속 쓸 때 3) 서버 재시작 없이 장시간 동작 후 첫 호출 시 |
| **재시도 후에도 계속 실패하는 경우** | 포트폴리오에 보유 종목이 여러 개일 때 **현재가를 동시에 조회**(Promise.all)하면서, 토큰 만료 시 **여러 요청이 동시에** 캐시를 비우고 **동시에** tokenP를 여러 번 호출합니다. 이때 KIS가 동일한(이미 만료된) 토큰을 다시 주거나 경쟁 상태가 나면, 재시도 요청까지 만료 토큰을 쓰게 되어 500이 반복됩니다. |

## KIS 토큰 규칙 (참고)

- 접근토큰 유효기간: **24시간** (1일 1회 발급 원칙)
- 갱신발급주기: **6시간** (6시간 이내는 기존 발급키로 응답)
- `access_token_token_expired` 값을 사용해 토큰을 관리

## 코드 측 조치 (이미 반영됨)

1. **메모이제이션·1일 1회 재발급**  
   `access_token_token_expired` 기준으로 토큰을 캐시하며, **유효기간이 지난 경우에만** tokenP를 호출해 1일 1회 재발급 원칙을 지킵니다.
2. **토큰 만료 시 자동 재시도**  
   inquire-price 등 호출 결과가 HTTP 500이고 `msg_cd: "EGW00123"` 또는 "만료된 token"이면, **캐시가 이미 만료된 경우에만** 캐시를 비우고 새 토큰을 발급한 뒤 한 번 재시도합니다. 캐시가 아직 유효하면 비우지 않아 불필요한 tokenP 호출을 막습니다.
3. **만료 시각 파싱**  
   KIS가 반환하는 `access_token_token_expired`를 파싱해 만료 1분 전에 재발급하도록 계산합니다.
4. **토큰 재발급 직렬화**  
   여러 종목 동시 조회 시 **한 번만** tokenP를 호출하고 나머지는 `refreshPromise`로 그 결과를 공유합니다.
5. **캐시 무효화 함수**  
   수동 재발급이 필요할 때만 `clearKisTokenCache()`를 호출합니다. 일상 동작에서는 유효기간 경과 시에만 재발급됩니다.

## 사용자가 할 수 있는 조치

- **별도 조치 없이** 한 번 더 요청(새로고침, 포트폴리오 다시 조회)하면, 위 자동 재시도 로직으로 새 토큰을 받아 호출이 성공할 수 있습니다.
- 계속 같은 오류가 나면 **KIS_APP_KEY**, **KIS_APP_SECRET**이 올바른지, KIS 개발자 포털에서 앱이 정상·만료되지 않았는지 확인하세요.
- 로컬 개발 시: 개발 서버를 **한 번 재시작**하면 캐시가 비워져 다음 호출에서 새 토큰이 발급됩니다.
